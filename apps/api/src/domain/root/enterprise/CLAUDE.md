# Enterprise

Domain entities and validators. This layer has **no dependencies on infrastructure** — no databases, no HTTP, no external libraries.

## Entities

Entities extend the abstract `Entity<Props>` base class from `@/core/entities/entity.ts`.

### Structure

```typescript
import { Entity } from '@/core/entities/entity.ts'
import type { UniqueEntityID } from '@/core/entities/unique-entity-id.ts'
import type { Optional } from '@/core/types/optional.ts'

export interface PostProps {
  title: string
  content: string
  createdAt: Date
  updatedAt: Date
}

export class Post extends Entity<PostProps> {
  // expose props through getters
  get title() { return this.props.title }
  get content() { return this.props.content }
  get createdAt() { return this.props.createdAt }
  get updatedAt() { return this.props.updatedAt }

  // mutable props use a setter + touch()
  set title(value: string) {
    this.props.title = value
    this.touch()
  }

  private touch() {
    this.props.updatedAt = new Date()
  }

  static create(
    props: Optional<PostProps, 'createdAt' | 'updatedAt'>,
    id?: UniqueEntityID,
  ) {
    return new Post(
      {
        ...props,
        createdAt: props.createdAt ?? new Date(),
        updatedAt: props.updatedAt ?? new Date(),
      },
      id,
    )
  }
}
```

### Conventions

- Always expose props via **getters** — never access `this.props` directly from outside.
- Mutable fields also have a **setter** that calls `touch()` to update `updatedAt`.
- `createdAt` and `updatedAt` are always present but **optional in `create()`** using `Optional<Props, 'createdAt' | 'updatedAt'>` — they default to `new Date()`.
- The static `create()` method is the only public constructor. It accepts an optional `id` to support rehydration from the database (pass the stored ID) or fresh creation (omit it, auto-generated by `UniqueEntityID`).
- Always call `id.toString()` when passing the ID outside the domain (e.g. returning from a use case).

### `UniqueEntityID`

Wraps the entity ID. Generates a `nanoid` by default or accepts an existing string value.

```typescript
import { UniqueEntityID } from '@/core/entities/unique-entity-id.ts'

const id = new UniqueEntityID()          // auto-generated
const id = new UniqueEntityID('abc-123') // rehydrated from DB
id.toString()                            // get the string value
id.equals(otherId)                       // compare two IDs
```

### `Optional<T, K>`

Utility type that makes specific keys optional. Used in `create()` signatures so callers don't need to supply `createdAt`/`updatedAt`.

```typescript
type Optional<T, K extends keyof T> = Omit<T, K> & Partial<Pick<T, K>>
```

### Existing Entities

| Entity | Mutable fields | Notes |
|--------|----------------|-------|
| `User` | — | `passwordHash` is read-only after creation; ID uses `crypto.randomUUID()` |
| `Folder` | `name` | Supports optional `parentId` for nesting |
| `Item` | `title`, `content` | Calls `validateTypeAndContent` in `create()` — may throw |

## Validators

Validators are pure functions that enforce business rules that depend on the combination of multiple fields. They **throw** (not return) errors, so callers must wrap them with `safeEither` when they need an `Either`.

### Structure

```typescript
import { SomeError } from './_errors/some-error.ts'

export function validateSomething(value: string): void {
  if (/* invalid */) {
    throw new SomeError('reason')
  }
}
```

### `safeEither`

Wraps a function that may throw and converts the result into `Either`. Used in use cases to integrate validators/entities that throw into the `Either` pipeline.

```typescript
import { safeEither } from '@/core/utils/safe-execute.ts'

const newItem = await safeEither(() => Item.create({ ... }))

if (newItem.isLeft()) {
  return left(newItem.value) // propagate the error
}

await repository.save(newItem.value)
```

### `validateTypeAndContent`

Validates that the `content` field is valid for the given `ItemType`. Called inside `Item.create()`.

| Type | Rules |
|---|---|
| `text` | Content is optional, no restrictions |
| `link` | Content required; must be a valid HTTPS URL; max 2048 chars |
| `document` | Content required; max 50,000 chars |
| `secret` | Content required; min 8 chars |

### Validator Errors

Validator errors live in `validators/_errors/` and follow the same convention as use case errors: extend `Error`, implement `BaseError`.

```typescript
import type { BaseError } from '@/core/errors/base-error.ts'

export class InvalidItemTypeError extends Error implements BaseError {
  constructor(message?: string) {
    super(message ?? 'Invalid item type')
  }
}
```

### Testing Validators

Validators are tested directly (no fakes or in-memory repos needed) using `toThrow` / `not.toThrow`.

```typescript
it('should throw for invalid url', () => {
  expect(() => validateTypeAndContent('link', 'not-a-url')).toThrow(InvalidItemTypeError)
})

it('should not throw for valid https url', () => {
  expect(() => validateTypeAndContent('link', 'https://example.com')).not.toThrow()
})
```
