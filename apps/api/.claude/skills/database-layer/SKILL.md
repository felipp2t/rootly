# Database Layer

Database configuration and schema definitions using Drizzle ORM.

## Drizzle Client

```typescript
import { db } from '@/infra/db/drizzle/index.ts'

// Query example
const users = await db.select().from(schema.users)
```

### Configuration

- Use `apps/api/docker-compose.yaml` for local Postgres + MinIO.
- Connection details are defined in `.env` (see `.env.example`).
- **snake_case** for database columns (camelCase in TypeScript).

## Creating New Schemas

1. Create a file in the `schema/` directory
2. Export it in `schema/index.ts`
3. Run `pnpm db:generate`
4. Run `pnpm db:migrate`

### Schema Example

```typescript
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core'
import { now } from '@/infra/db/drizzle/helpers/now.ts'
import { nanoid } from '@/infra/lib/nanoid.ts'

export const users = pgTable('users', {
  id: text('id')
    .primaryKey()
    .$defaultFn(() => nanoid()),
  name: text('name').notNull(),
  email: text('email').notNull(),
  passwordHash: text('password_hash').notNull(),
  createdAt: timestamp('created_at', { withTimezone: true })
    .defaultNow()
    .notNull(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).$onUpdate(now),
})
```

## Helpers

### `nanoid`

Used to generate unique IDs for primary keys. Uses a custom alphanumeric alphabet (`A-Z`, `a-z`,
`0-9`) with 12 characters.

```typescript
import { nanoid } from '@/infra/db/drizzle/helpers/nanoid.ts'

id: text('id')
  .primaryKey()
  .$defaultFn(() => nanoid()),
```

### `now`

Used to set `updatedAt` to the current database timestamp on every update via `$onUpdate`.

```typescript
import { now } from '@/infra/db/drizzle/helpers/now.ts'

updatedAt: timestamp('updated_at', { withTimezone: true }).$onUpdate(now),
```

## Schema Conventions

### Naming

- Table names must always be **plural** (e.g., `users`, `folders`, `items`).
- TypeScript property names use **camelCase**; database column names use **snake_case**.

```typescript
// correct
export const folders = pgTable('folders', {
  folderId: text('folder_id').primaryKey(),
  createdAt: timestamp('created_at').notNull(),
})

// wrong
export const folder = pgTable('folder', {
  folderId: text('folderId').primaryKey(), // column name must be snake_case
})
```

### Standard Columns

All tables except `users` must include:

```typescript
{
  id: text('id')
    .primaryKey()
    .$defaultFn(() => nanoid()),
  createdAt: timestamp('created_at', { withTimezone: true })
    .defaultNow()
    .notNull(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).$onUpdate(now),
}
```

### Enums

```typescript
import { pgEnum } from 'drizzle-orm/pg-core'

export const itemTypeEnum = pgEnum('item_type', [
  'link',
  'document',
  'secret',
  'text',
])

// In table
type: itemTypeEnum('type').notNull(),
```

Adding a new value to an existing `pgEnum` is a simple additive migration (`ALTER TYPE ... ADD
VALUE`). **Removing** a value is not directly supported by Postgres (`DROP VALUE` doesn't exist)
— `drizzle-kit generate` handles this by emitting a second migration that recreates the enum type
(`ALTER COLUMN ... SET DATA TYPE text` → `DROP TYPE` → `CREATE TYPE` → cast back). This is normal,
expected `drizzle-kit` behavior, not a mistake to work around.

## Migrations

Migrations are automatically generated by `drizzle-kit`.

- **NEVER modify any file inside `apps/api/drizzle/`** — they are auto-generated and manual
  changes may cause issues.

### After Modifying Schemas

```bash
cd apps/api
pnpm db:generate
pnpm db:migrate
```

## Query Examples

### Select

```typescript
import { schema } from 'path/to/schema/index.ts'

const users = await db.select().from(schema.users)

const user = await db.select().from(schema.users).where(eq(schema.users.email, 'test@example.com'))
```

### Insert

```typescript
const [user] = await db.insert(schema.users).values({ name: 'John Doe' }).returning()
```

### Delete

```typescript
await db.delete(schema.users).where(eq(schema.users.id, userId))
```

## Repositories

- Every `DrizzleXxxRepository`'s `create`/`save`/`delete` methods must call
  `DomainEvents.dispatchEventsForAggregate(entity.id)` (or `new UniqueEntityID(id)` for `delete`,
  which only has the raw id) after the write succeeds — this is what actually fires any domain
  events the entity queued up via `addDomainEvent`. It's easy to add a new repository and forget
  this; always check it explicitly when adding one.
