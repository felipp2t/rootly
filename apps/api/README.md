# Rootly API

REST API for Rootly, built with Fastify, Drizzle ORM and PostgreSQL, following Clean Architecture + DDD.

## Stack

| Concern | Technology |
|---|---|
| HTTP framework | Fastify 5 |
| ORM | Drizzle ORM |
| Database | PostgreSQL |
| Object storage | MinIO (S3-compatible) |
| Validation | Zod 4 |
| Auth | JWT (`jose`) + Argon2 password hashing |
| Real-time | WebSockets (`@fastify/websocket`) |
| API docs | Swagger / OpenAPI, served via `@fastify/swagger-ui` |
| Testing | Vitest (unit) + Testcontainers (e2e, real PostgreSQL) |
| Linting/formatting | Biome |

## Architecture

Clean Architecture + DDD, in three layers:

```
src/
├── core/        # Framework-agnostic primitives: Entity, AggregateRoot, Either, DomainEvents
├── domain/       # Business logic, split by bounded context (root, activity, notification)
│   └── <context>/
│       ├── enterprise/    # Entities, domain events, validators — no infra dependencies
│       └── application/   # Use cases, repository interfaces, subscribers
└── infra/       # Fastify controllers, Drizzle repositories, factories (dependency wiring)
```

Every mutation goes through a use case that returns `Either<Error, Result>` (never throws), and
side effects (like recording an activity log entry) are modeled as domain events dispatched after
the write succeeds — see `src/domain/activity` for the audit-log subscribers.

## Features

- **Auth** — register/login/refresh via JWT stored in HTTP-only cookies, Argon2 password hashing
- **Workspaces** — multi-tenant workspaces with invites and a granular RBAC system (custom roles,
  per-resource/action permissions)
- **Folders & items** — hierarchical folders containing links, documents, secrets and text notes
- **File upload** — document items can be uploaded as files, stored in MinIO
- **Activity log** — full audit trail of who did what and when, across folders, items, members,
  workspaces and roles
- **Real-time notifications** — pushed over WebSocket as they happen
- **OpenAPI docs** — interactive Swagger UI, always in sync with the Zod route schemas

## Getting started

```bash
# from the repo root
pnpm install

cd apps/api
cp .env.example .env   # fill in JWT_SECRET at minimum

# start Postgres + MinIO
docker compose up -d

# apply migrations
pnpm db:migrate

# optional: populate demo data (30 users, 4 workspaces, roles, folders, items)
# password for every seeded user is `password123`, emails are printed at the end
pnpm db:seed

pnpm dev
```

The API listens on `http://localhost:3333`. Interactive API docs are at `http://localhost:3333/docs`.

## Commands

```bash
pnpm dev              # start with hot reload
pnpm build            # type-check and compile to dist/
pnpm lint             # Biome check
pnpm typecheck        # tsc --noEmit
pnpm format           # Biome format --write

pnpm test             # unit tests (in-memory repositories, no DB required)
pnpm test:e2e         # e2e tests (spins up a real Postgres via Testcontainers)
pnpm test:cov         # unit tests with coverage report

pnpm db:generate      # generate a migration after changing a schema file
pnpm db:migrate       # apply pending migrations
pnpm db:studio        # open Drizzle Studio
pnpm db:seed          # reset and repopulate the database with demo data

pnpm openapi          # regenerate openapi.json from the route schemas
```

> Files under `drizzle/` are auto-generated by `drizzle-kit` — never edit them by hand.

## Testing conventions

- Use cases are tested against in-memory repository fakes (`test/repositories/`) — no database
  needed, runs in milliseconds.
- Controllers have a matching `*.e2e.spec.ts` that hits the real Fastify app + a disposable
  Postgres container, covering the happy path only (error branches are covered in the unit specs).
- Domain event subscribers are tested by dispatching the event directly and flushing the
  microtask queue, since `DomainEvents.dispatch()` doesn't await handlers.
