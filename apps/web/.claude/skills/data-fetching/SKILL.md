# Data Fetching

## The generated client

`src/api/` is generated by `orval` from the API's OpenAPI spec — never edit it by hand. Every
endpoint gets a `useXxx` (query) or `useXxx` (mutation) hook, a `useXxxSuspense` variant, and a
`getXxxQueryKey()` helper, all built on TanStack Query.

```bash
pnpm generate   # regenerates openapi.json on the API, then the client here
```

Run this after changing any route on the API side. See the `routing-pages`/`monorepo-overview`
skills for related commands.

## Responses are status-discriminated, not thrown

Every generated hook resolves to `{ data, status }`, one shape per possible HTTP response — mirroring
the API's `Either` pattern (see the API's `use-cases` skill). **Never wrap these calls in
try/catch** — check `status` instead:

```typescript
const { data: meRes } = useGetMeSuspense()
const user = meRes.status === 200 ? meRes.data : null
```

For mutations, branch on the resolved response's `status` to decide the UI outcome (success
toast, validation error, generic failure) — see the Mutations section below.

## The Suspense / Loader / Skeleton pattern

Any component that fetches data follows this three-part shape: an outer component wrapping a
`Suspense` boundary, an inner `*Loader` component that actually calls the `useXxxSuspense` hook,
and a `*Skeleton` fallback:

```typescript
export function ActivitySection({ workspaceId }: { workspaceId: string }) {
  return (
    <Suspense fallback={<ActivitySectionSkeleton />}>
      <ActivitySectionLoader workspaceId={workspaceId} />
    </Suspense>
  )
}

function ActivitySectionLoader({ workspaceId }: { workspaceId: string }) {
  const { data: activityRes } = useGetActivityLogsSuspense(workspaceId, {})
  // ... branch on activityRes.status, render the real content
}

export function ActivitySectionSkeleton() {
  // ... Skeleton placeholders matching the real layout's shape
}
```

This keeps loading states declarative (via `Suspense`) while still being able to render a
layout-accurate skeleton rather than a generic spinner. Export the `*Skeleton` too if a parent
route wants to use it as its own `pendingComponent`.

## Mutations

Call the generated mutation hook directly — don't reach for a raw `useMutation`. Handle the
outcome by checking `response.status`, then invalidate the relevant query and give feedback via
`sonner`'s `toast`:

```typescript
const queryClient = useQueryClient()
const updateProfileMutation = useUpdateProfile()

async function onSubmit(name: string) {
  const response = await updateProfileMutation.mutateAsync({ data: { name } })

  if (response.status === 204) {
    queryClient.invalidateQueries({ queryKey: getGetMeQueryKey() })
    toast.success('Profile updated successfully')
  } else if (response.status === 400) {
    toast.error('Invalid name. Must be at least 3 characters.')
  } else {
    toast.error('Failed to update profile. Please try again later.')
  }
}
```

Use `mutation.isPending` to disable the submit control / show a spinner.

## Auth (`src/lib/auth.tsx`)

`AuthProvider` wraps the app and exposes `useAuth()` → `{ isAuthenticated, setAuthenticated, user }`,
consumed by the router's `beforeLoad` guard (see the `routing-pages` skill). It reads `/me` via
`useGetMe` with `retry: false, staleTime: Infinity`, and keeps an `optimisticAuth` flag so a
fresh login flips `isAuthenticated` synchronously instead of racing the post-login refetch.

## RBAC permission checks (`src/hooks/use-workspace-permissions.ts`)

`useWorkspacePermissions(workspaceId)` fetches the caller's resolved permissions for that
workspace and returns a `can(resource, action)` predicate — mirrors the API's RBAC model
(`resource:action`, where a role's `action: 'all'` implicitly grants every action on that
resource):

```typescript
const { can } = useWorkspacePermissions(workspaceId)

if (!can('member', 'invite')) return null
```

Use this to conditionally render actions/sections the current user isn't allowed to perform,
consistent with what the API would reject with a `403`.
